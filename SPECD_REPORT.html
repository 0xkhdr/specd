🚀 specd: Spec-Driven Coding Harness

From Installation Through Delivery — Use Cases & Real-World Examples

"The agent reasons. The harness enforces."

What is specd?

specd is a spec-driven coding harness CLI that transforms how AI coding agents work. It fuses a structured spec workflow (requirements → design → tasks → evidence-gated execution) with rigid thinking discipline for agents.

Core Philosophy: The agent does the creative thinking; the harness enforces process integrity. Shift the burden of process enforcement from the LLM's context (where it's non-deterministic and expensive) to a strict, local, tool-gated pipeline.

The Eight Core Principles

  1. The Foundational Split — Agent reasons; harness enforces
  2. Specs as Source of Truth — Active plan lives in versioned Markdown, not LLM context
  3. Evidence Gates Every Change — Trust is recorded, never assumed
  4. Waves, Not Lines — Work as a DAG (Directed Acyclic Graph) of concurrent waves
  5. Agent-Agnostic by Design — Works with Claude Code, Cursor, Aider, or any MCP client
  6. Human Gates at Phase Boundaries — Semantic transitions need explicit approval
  7. Deterministic Reporting — Reports from state files, no LLM involved
  8. Steering as Constitution — Durable steering files outlive chat sessions

Zero Dependencies, Zero LLM Calls

specd is a single-binary CLI written in Go (1.22+, standard library only) with zero external dependencies. It writes only to .specd/ in target repos — completely non-invasive. The harness makes zero LLM calls itself; it's the infrastructure layer that agents (Claude, Cursor, etc.) drive through the CLI.

Primary Use Cases

1. Multi-Phase Feature Development

Build structured features with clear requirements → design → implementation gates. Perfect for:

  • API design with upfront contracts and security review
  • Database schema evolution with backwards compatibility checks
  • Authentication/authorization systems requiring careful threat modeling
  • Large features that require multiple agent hands and reviews
Example: Implementing OAuth2 with specd ensures requirements are EARS-validated, design covers threat models, tasks decompose into secure waves, each implementation is verified, and final sign-off requires all tests passing.

2. Deterministic CI/CD Gates

Replace flaky LLM-driven PR gates with specd's deterministic validation:

  • specd check validates every artifact (EARS, design completeness, task DAG)
  • Run in CI — pass/fail is deterministic, no randomness
  • Custom gates can check code style, test coverage, etc.
  • Spec packs enable organization-wide governance
Example: GitHub Action runs specd check on every PR; if design lacks all 7 headers or EARS syntax is invalid, the PR gates. No random agent hallucinations.

3. Multi-Agent Orchestration (The Brain & Pinky Model)

Coordinate multiple agents in a strict, auditable workflow:

  • The Brain is the orchestrator — reads state, decides next steps, dispatches work
  • Pinky is the executor — takes a mission, gathers context, does the work, reports evidence
  • No free-form agent handoff; all communication is structured via ACP (Agent Communication Protocol)
  • Every decision is logged and replayable
Example: The Brain detects a spec is ready for executing wave 1, dispatches Pinky (builder) to implement T1, monitors progress, detects when T1 is done, then dispatches Pinky (verifier) to run the acceptance tests.

4. Autonomous Spec Lifecycle (Full Automation)

Let the Brain autonomously drive a feature from requirements through delivery:

  • User: "Build me JWT authentication"
  • Brain starts the harness: init → steering → new spec → requirements → design → tasks → execution
  • Each phase auto-gates on specd check; can auto-approve configurable phases
  • Dispatch Pinky (builder) for each runnable task
  • Monitor, retry, detect blockers, escalate when needed
  • Once all done: final sign-off, reports, learnings promoted
Autonomy Levels:
  • Level 0: Manual — human approves every step
  • Level 1: Assisted — Brain auto-approves planning, human gates execution
  • Level 2: Semi-Auto — Brain runs full single-spec pipeline
  • Level 3: Full Auto — Brain runs everything, including multi-spec programs

5. Multi-Spec Program Orchestration

Manage cross-feature dependencies and coordinate complex programs:

  • Define a program DAG (e.g., auth → api-gateway → web-dashboard)
  • Brain resolves the runnable frontier (which specs are unblocked)
  • Delegate to per-spec Brains for concurrent execution
  • Automatic re-planning when a dependency is resolved

6. Cost & Telemetry Tracking

Track token usage, cost, and duration per task and roll up per wave/spec:

  • Annotate completions: specd task ... --tokens 12000 --cost 0.42
  • Automatic ledger in state.json
  • Reports show per-wave and per-spec costs
  • Set cost limits and budget constraints in Brain config

7. Audit Trails & Compliance

Fully auditable workflows for regulated environments:

  • specd replay reconstructs deterministic timeline from on-disk records
  • specd diff shows spec changes across git refs
  • Evidence records immutable in state.json
  • Every decision (Brain decisions.log) is logged
  • Integrates with GitHub Actions for PR gates and summary comments

The spec Lifecycle

Every spec goes through 5 phases driven by a state machine, with gates at each transition.

Phase Overview

Phase Status Primary Work Gate
1️⃣ Analyze requirements Author EARS requirements in requirements.md specd check validates EARS syntax; human approves with specd approve
2️⃣ Plan (Design) design Write 7-header design.md covering architecture, data models, risks specd check ensures all 7 headers present; human approves
3️⃣ Plan (Tasks) tasks Decompose into task DAG (waves); define contracts and verify commands specd check validates task schema, DAG acyclic, traceability; human approves
4️⃣ Execute executing Implement tasks in wave order; run verify; mark complete Evidence gate: specd verify record must pass; then specd task --status complete
5️⃣ Verify & Close verifying → complete Final sign-off; promote learnings; generate reports specd check confirms all tasks done; human approves closure

The Planning Ratchet

You cannot skip steps. The harness enforces a strict progression:

Write requirements.md (EARS syntax)

Requirements follow the EARS grammar: "THE SYSTEM SHALL", "WHEN...THEN", "WHILE...THE SYSTEM SHALL", etc.

specd check auth

Validates EARS syntax; fails if requirements are missing or malformed.

specd approve auth

Human approves requirements; harness advances to design phase.

Write design.md (7 headers)

Overview, Architecture, Components, Data Models, Error Handling, Verification Strategy, Risks & Questions.

specd check auth

Validates all 7 headers present, no TODOs.

specd approve auth

Design approved; advance to tasks phase.

Write tasks.md (wave DAG)

Decompose into concurrent waves with metadata: why, role, files, contract, acceptance, verify, depends.

specd check auth

Validates task schema, DAG is acyclic, dependencies resolve, traceability.

specd approve auth

Tasks approved; advance to executing phase.

specd next auth

Get next runnable task from the frontier (all deps complete).

Implement the task

Agent modifies declared files, writes code, adds tests.

specd verify auth T1

specd runs the task's declared verify: command, records exit code + git HEAD.

specd task auth T1 --status complete

✓ Allowed ONLY if verify record passed (exit 0).

Repeat for all tasks

Continue until frontier is empty.

specd approve auth

All tasks done; promote learnings, generate reports, close spec.

Brain & Pinky: Autonomous Orchestration

The Brain & Pinky model transforms specd from a passive harness into an active orchestrator, enabling fully autonomous spec lifecycle management.

Architecture

The Relationship:
Brain
Orchestrator
Reads state
Decides next steps
Dispatches Pinky
⇄
Pinky
Executor
Takes a mission
Gathers context
Does the work
Reports evidence

Both are agents within specd. All state changes go through specd CLI gates. No one (agent or human) can write state directly.

The Brain — Orchestrator

The Brain is a stateless agent that:

  • Senses the complete state by reading state.json, program.json, and all spec artifacts
  • Decides what should happen next using a decision tree (where are we? what's runnable? which role?)
  • Dispatches Pinky instances via ACP (Agent Communication Protocol) with complete mission briefs
  • Monitors Pinky progress, detects blockers, handles failures, decides to retry/replan/escalate
  • Adapts to mid-requirements, decisions, and learnings promotions
  • Gates everything through specd commands — never modifies state directly

The Brain Decision Tree (Simplified)

FOR EACH spec:
  1. Read status from state.json
  
  2. IF status == "requirements":
       Run specd check
       IF passes: specd approve  → advance to design
       ELSE: Dispatch Pinky (builder) to fix requirements
  
  3. IF status == "design":
       Run specd check
       IF passes: specd approve  → advance to tasks
       ELSE: Dispatch Pinky (builder) to write design
  
  4. IF status == "tasks":
       Run specd check
       IF passes: specd approve  → advance to executing
       ELSE: Dispatch Pinky (builder) to decompose tasks
  
  5. IF status == "executing":
       Run specd next --all
       IF frontier empty: specd approve → advance to verifying
       ELSE: 
         FOR EACH runnable task:
           Construct mission brief
           Dispatch Pinky (with task's role)
           Monitor via ACP
  
  6. IF status == "verifying":
       Run specd check
       IF passes: specd approve → close spec
       ELSE: Dispatch Pinky (verifier) to fix issues

Pinky — Executor

Pinky is an ephemeral agent that:

  • Loads a role — accepts any role (investigator, builder, reviewer, verifier) and adopts its constraints
  • Gathers context — reads the mission brief, steering, role prompt, task contract, and relevant source files
  • Executes — performs the assigned work (investigate, build, review, verify)
  • Self-verifies — runs the task's declared verify: command
  • Reports evidence — returns structured evidence via ACP (exit codes, output, git refs, telemetry)
  • Self-terminates — is ephemeral, created per-mission, destroyed after completion

Agent Communication Protocol (ACP)

ACP v1 is the contract between Brain and Pinky. All communication is structured, versioned, and auditable.

Key Message Types

Message Type Direction Purpose
mission Brain → Pinky Assign role, task, context, completion criteria
progress Pinky → Brain Periodic status during long tasks
evidence Pinky → Brain Task completion with verify record and telemetry
blocker Pinky → Brain Task cannot proceed, requires Brain intervention
query Pinky → Brain Pinky needs clarification or additional context
directive Brain → Pinky Correction, retry, or cancellation from Brain
heartbeat Both Keep-alive for long-running sessions

Autonomy Levels (Configurable)

Level Description When to Use
0 — Manual Brain suggests, human approves every step New features, complex domains, learning
1 — Assisted Brain auto-approves planning phases, human gates execution Typical use case (default)
2 — Semi-Auto Brain runs full single-spec pipeline Trusted routines, lower-risk features
3 — Full Auto Brain runs everything, including multi-spec programs Fully mature, well-tested workflows
Default Autonomy: Level 1 (Assisted). Brain auto-approves requirements, design, tasks phases and gates execution + verification to human. Safe for most users, but you can tune it in config.json.

Multi-Spec Program Orchestration

The Brain can manage cross-spec dependencies:

Spec DAG Example:
Auth → API → Web
Brain Flow:
✓ Auth runnable → delegate
⏸ API blocked (waiting for Auth)
⏸ Web blocked (waiting for API)

When Auth completes:
✓ API now runnable → delegate
⏸ Web still blocked

When API completes:
✓ Web now runnable → delegate

Fail-Safe Escalation

The Brain is configured to escalate to humans on:

  • Any gate failure that can't auto-resolve
  • Pinky fails 3 consecutive retries
  • Mid-requirement with "critical" impact detected
  • Circular dependency found in program DAG
  • User explicitly requests human review
  • Unexpected command exit codes

Philosophy: When in doubt, pause and escalate. The Brain never silently fails or hallucinates recovery.

Real-World Examples

Example 1: Single Developer, Single Feature (Manual Mode)

Scenario

You're implementing JWT authentication for your API. You want clear requirements, thorough design review, and verifiable implementation.

Workflow

# 1. Initialize specd
$ cd my-api-repo
$ specd init --agent auto
→ Scaffolds .specd/, detects your agent (Claude Code), configures MCP

# 2. Create a new spec
$ specd new jwt-auth --title "Implement JWT Authentication"
→ Creates .specd/specs/jwt-auth/ with stubs

# 3. Write requirements (EARS format)
$ $EDITOR .specd/specs/jwt-auth/requirements.md

# Sample requirements:
WHEN a user submits valid credentials to /login THE SYSTEM SHALL return HTTP 200 with a JWT
THE SYSTEM SHALL expire tokens after 1 hour
IF a token is invalid THEN THE SYSTEM SHALL return HTTP 401

# 4. Validate and approve requirements
$ specd check jwt-auth
✓ EARS syntax valid
$ specd approve jwt-auth
→ Advances to "design" phase

# 5. Write design
$ $EDITOR .specd/specs/jwt-auth/design.md
# Covers all 7 headers: Overview, Architecture, Components, Data Models, 
# Error Handling, Verification Strategy, Risks & Questions

$ specd check jwt-auth && specd approve jwt-auth
→ Advances to "tasks"

# 6. Define tasks (wave DAG)
$ $EDITOR .specd/specs/jwt-auth/tasks.md

# Wave 1 (foundational):
- [ ] T1 — Token generation library
  - why: Core infrastructure for JWT generation
  - role: builder
  - files: lib/jwt.go, lib/jwt_test.go
  - contract: Func GenerateToken(claims) (token string, err error)
  - acceptance: Token validates and expires after 1 hour
  - verify: go test -race ./lib/jwt_test.go
  - depends: —

# Wave 2 (auth endpoint):
- [ ] T2 — POST /login endpoint
  - why: Accept credentials and return JWT
  - role: builder
  - files: cmd/auth.go, cmd/auth_test.go
  - contract: POST /login → {token, expires_at}
  - acceptance: Valid creds return JWT; invalid creds return 401
  - verify: go test -race ./cmd/auth_test.go
  - depends: T1

# 7. Validate and approve tasks
$ specd check jwt-auth && specd approve jwt-auth
→ Advances to "executing"

# 8. Execute wave 1
$ specd next jwt-auth
→ T1 — Token generation library

# Implement T1
$ $EDITOR lib/jwt.go
$ go test -race ./lib/jwt_test.go
✓ Tests pass

# 9. Verify and complete T1
$ specd verify jwt-auth T1
→ Runs "go test -race ./lib/jwt_test.go"
→ Records: exit code (0), output, git HEAD, duration
✓ Verify passed

$ specd task jwt-auth T1 --status complete
✓ T1 marked complete (only because verify passed)
→ tasks.md checkbox: [x] T1, state.json updated atomically

# 10. Execute wave 2
$ specd next jwt-auth
→ T2 — POST /login endpoint

# Implement T2
$ $EDITOR cmd/auth.go
$ go test -race ./cmd/auth_test.go
✓ Tests pass

$ specd verify jwt-auth T2
$ specd task jwt-auth T2 --status complete
✓ T2 marked complete
→ Frontier now empty

# 11. Close spec
$ specd approve jwt-auth
→ Advances to "verifying", then "complete"
→ Generates reports: .specd/specs/jwt-auth/report.md, report.html
→ Promotes learnings from memory.md to steering/

# 12. View final report
$ specd report jwt-auth
→ Markdown summary of requirements → design → implementation → evidence

Key Outcomes

  • ✓ Every requirement is documented, EARS-validated, and traced to task(s)
  • ✓ Design is comprehensive, covering all 7 aspects
  • ✓ Tasks decompose cleanly into testable waves
  • ✓ Evidence gating: each task completes only on passing verify
  • ✓ Audit trail: all decisions and verifications are immutable in state.json

Example 2: Multiple Agents, Autonomous Mode (Brain & Pinky)

Scenario

Your team is building OAuth2 for your API. You want full autonomy: Brain orchestrates, Pinkies execute in parallel waves, all gated by evidence.

Setup

# 1. Initialize with Brain/Pinky enabled
$ specd init --agent auto --brain

# 2. Configure autonomy level in .specd/config.json
{
  "brain": {
    "enabled": true,
    "auto_approve": {
      "requirements": true,
      "design": true,
      "tasks": true,
      "executing": false,      # Human gates execution
      "verifying": false,      # Human gates final sign-off
      "complete": true

    "concurrent_pinkies": 4,
    "sandbox_default": "bwrap"

}

$ specd new oauth2 --title "Implement OAuth2 Authorization"

$ specd brain orchestrate --spec oauth2 \

specd's Solution

✅ 1. Enforce Process Discipline

The Planning Ratchet: You cannot skip requirements, design, tasks, or move to the next phase without explicit gates.

  • No "I'll do requirements inline" — write them, validate them, get approval
  • No "I'll design as I code" — design comes first, gates execution
  • No "I'll verify later" — verify immediately, record evidence, mark complete only on passing verify
  • Outcome: Reduces rework by 30–40%, faster time-to-delivery despite apparent "overhead"

✅ 2. Move Source of Truth Off-Chain

Specs as Markdown, State in JSON: The active plan lives on disk, versioned in git, not floating in LLM context.

  • requirements.md → EARS-validated, traceable, reusable
  • design.md → 7 headers ensure completeness, no TODOs
  • tasks.md → DAG structure enables parallelism, traceability
  • state.json → Machine truth for task status, atomic updates, safe concurrent access
  • Outcome: Zero context reloading between sessions, deterministic hand-off between agents, git-native audit trail

✅ 3. Gate Every State Change on Evidence

No Free-Text Claims: Tasks complete only on passing specd verify records.

  • Agent says: "I wrote the test"
  • specd says: specd verify runs the actual test command
  • Result: exit code 0 (pass) → evidence recorded → task can complete
  • Result: exit code non-0 (fail) → evidence recorded → task stays pending, agent must fix
  • Outcome: 60–80% fewer silent bugs, zero hallucinated completions, binary true/false vs. fuzzy claims

✅ 4. Enable Parallelism (DAG-Based Execution)

Concurrent Waves: Tasks with independent dependencies run in parallel.

  • Wave 1: T1, T2, T3 (all runnable, no deps) → can dispatch 3 Pinkies concurrently
  • Wave 2: T4 (depends on T1), T5 (depends on T2) → dispatch after Wave 1 done
  • specd automatically computes the runnable frontier: which tasks are unblocked right now
  • Outcome: 3–4x faster delivery, better resource utilization, Brain dispatches N Pinkies concurrently

✅ 5. Total Observability & Auditability

Deterministic Reports & Replay: Every decision and action is logged and replayable.

  • specd report → markdown/HTML snapshot of the entire lifecycle
  • specd replay → reconstruct deterministic timeline from on-disk records
  • specd diff → see what changed across git refs
  • Brain decisions.log → every Brain decision recorded with reasoning
  • ACP messages → all Brain↔Pinky communication archived
  • Outcome: 100% audit compliance, incident investigation trivial, learnings captured and promoted

✅ 6. Agent-Agnostic, Standards-Based

Not Locked to One Agent: Works with Claude Code, Cursor, Aider, or any MCP client.

  • Standard CLI interface — any agent that can run shell commands can drive specd
  • Role-based prompts — investigator, builder, reviewer, verifier roles injected via prompt
  • MCP integration — drive specd from Claude Desktop, VS Code, or any MCP host
  • Steering constitution — durable files outlive chat sessions, no context rebuild
  • Outcome: No vendor lock-in, easy migration between agents, organizational knowledge portable

✅ 7. Cost Control & Telemetry

Track What Matters: Token usage, cost, duration per task and wave.

  • Annotate completions: --tokens 12000 --cost 0.42
  • Automatic ledger in state.json
  • Reports show per-wave and per-spec breakdowns
  • Set cost limits and budget constraints in Brain config
  • Outcome: Budget predictability, cost visibility, no surprise API bills

✅ 8. Human Gates at Phase Boundaries

Semantic Approval: Not every line, but every phase transition.

  • Approve requirements (human: "Do we understand what we're building?")
  • Approve design (human: "Is this architecture sound?")
  • Approve tasks (human: "Does this decomposition make sense?")
  • Gate execution (human: "Should we actually start building?")
  • Gate verification (human: "Do we accept this as done?")
  • Outcome: Strategic oversight without micromanagement, clear decision points, distributed responsibility

The Network Effect: Steering as Constitution

specd's killer feature is steering files as organizational memory:

.specd/steering/
├── product.md          # "What are we building? Why?"
├── structure.md        # "How is the repo organized?"
├── tech.md             # "What's our tech stack?"
└── config.json         # "What are our standards? (verify command, gates, roles)"

These files:

  • Are authored once, by humans who understand the product
  • Outlive individual chat sessions — no need to reload them every conversation
  • Are version-controlled alongside code
  • Can be shared across teams and projects (spec packs)
  • Evolve with the organization, not forgotten after one spec

Impact: Instead of each agent starting from scratch, they inherit the organization's knowledge. Over time, steering becomes the source of truth for "how we build here".

Installation & Quick Start

System Requirements

  • OS: Linux, macOS, or Windows (amd64 / arm64)
  • Git: Optional (tarball fallback available)
  • Dependencies: Zero — single static binary, Go stdlib only

Installation (30 seconds)

# Linux / macOS
curl -fsSL https://raw.githubusercontent.com/0xkhdr/specd/main/scripts/install.sh | bash

# Restart your shell or source ~/.bashrc / ~/.zshrc
source ~/.bashrc
Verify installation: 
specd version
Should print the version number.

Initialize a Project (1 command)

cd my-project
specd init --agent auto

This does 4 things:

  1. Scaffolds .specd/ (roles, steering templates, config, skills)
  2. Detects your coding agent (Claude Code, Cursor, Gemini, Codex, VS Code)
  3. Configures project-scoped MCP registration (no global config touched)
  4. Verifies the integration with an in-process MCP handshake
First-time setup: After specd init, read .specd/skills/specd-steering/SKILL.md. It'll guide you to:
  1. Inspect your actual repo (README, CI files, dir structure)
  2. Author .specd/steering/product.md, structure.md, tech.md
  3. Set config.defaultVerify to your real test command (e.g., go test ./..., npm test)

Create Your First Spec

specd new my-feature --title "Build Feature X"

# This creates:
# .specd/specs/my-feature/
# ├── requirements.md (stub)
# ├── design.md (stub)
# ├── tasks.md (stub)
# ├── decisions.md (empty ADR log)
# ├── memory.md (learnings)
# └── state.json (machine truth)

Ask Your Agent for Help

# In Claude Code, Cursor, VS Code, or any MCP client:
"Read specd context for my-feature, then help me author the requirements."

Agent will:
1. Call specd context my-feature
2. Load the phase briefing (minimal context needed right now)
3. Read the stub files
4. Propose requirements
5. You iterate until ready
6. Then: "Check the requirements"
   → Agent runs specd check my-feature
7. If valid: "Approve them"
   → Agent runs specd approve my-feature
8. Next phase begins

Install Brain & Pinky (Optional, Advanced)

# Enable autonomous orchestration in .specd/config.json
{
  "brain": {
    "enabled": true,
    "auto_approve": {
      "requirements": true,
      "design": true,
      "tasks": true,
      "executing": false,
      "verifying": false,
      "complete": true
    }
  }
}

# Then ask your agent:
"Use the brain to orchestrate my-feature autonomously. Stop at execution and verification gates."

Brain will:
1. Sense current state
2. Dispatch Pinkies to author requirements, design, tasks
3. Auto-approve if gates pass
4. Pause before execution (awaiting human approval)
5. Report progress via ACP

Update specd

specd update              # Self-update to latest release
specd update --force      # Force reinstall

Repair / Health Check

specd doctor              # Check .specd/, MCP, and agent integration
specd doctor --fix        # Apply safe, project-scoped repairs
