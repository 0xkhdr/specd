# Prompt: Full Regression & Ship Readiness Review — specd CLI

> **Context:** You are a senior coding agent (Claude Code) working on the `specd` project — a spec-driven coding harness CLI written in Go (stdlib only, zero runtime dependencies). The project is at `https://github.com/0xkhdr/specd`.
>
> **Current State:** 7 specs have been implemented under the `spcs/` directory. The tool is considered feature-complete and ready for final review before shipping. You must perform a comprehensive regression to validate correctness, completeness, and ship readiness.
>
> **Goal:** Create an `ACTION_PLAN.md` in the project root, then execute it stage-by-stage. After each stage, commit and push before proceeding to the next.

---

## 0. Pre-Flight: Self-Discovery & Action Plan Creation

1.  **Clone/refresh the repo** if not already present, and `cd` into it.
2.  **Search and read** these files to build your own understanding:

    *   `TESTING.md` — test strategy, coverage policy, determinism invariants, CI matrix.
    *   `docs/validation-gates.md` — the 7 spec gates + 2 repo-global gates.
    *   `docs/contributor-guide.md` — codebase map, key contracts, extension recipes, style invariants.
    *   `docs/user-guide.md` — spec lifecycle, artifact formats, task execution flow.
    *   `docs/command-reference.md` — every command, flag, exit code, env var.
3.  **Inspect the `spcs/` directory** — list all 7 specs, read their `requirements.md`, `design.md`, and `tasks.md` to understand what was implemented.

## Execution Rules

1.  **Work one stage at a time.** Do not start Stage N+1 until Stage N is fully complete.
2.  **After each stage:**
    *   Mark all items in that stage as `[x]` done in `ACTION_PLAN.md`.
    *   `git add -A && git commit -m "regression: stage N — <description>"`
    *   `git push`
3.  **If a stage fails:**
    *   Do not proceed to the next stage.
    *   Fix the failure within the stage.
    *   Re-run the stage checklist.
    *   Commit and push the fix with a descriptive message.
    *   Only then proceed.
4.  **If you discover a bug:**
    *   Fix it immediately if it's a blocker.
    *   Document it in `REGRESSION_FINDINGS.md` if it's a warning or info item.
    *   Re-run affected tests after any fix.
5.  **Use `SPECD_JSON=1` or `--json`** when parsing `specd` output programmatically.
6.  **Do not hand-edit `state.json` or `.specd/` files directly** — always use `specd` CLI commands.
7.  **Preserve determinism:** never introduce map-iteration-order dependencies or wall-clock dependencies in tests.
