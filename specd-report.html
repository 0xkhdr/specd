1 · What specd is

Most AI coding today is free-form prompting: the plan lives inside the model's context window, "done" means the model said so, and process discipline is re-negotiated every session. specd inverts that. It moves process enforcement out of the LLM's non-deterministic context and into a strict, local, tool-gated pipeline. The plan lives as versioned Markdown on disk; every state change requires recorded proof; work is a DAG of concurrent waves, not a flat todo list.

│  AGENT  (LLM)   │        │ HARNESS (specd) │

🔄

Planning Ratchet

Human-approved phase gates: Analyze → Plan → Execute → Verify → Reflect. No skipping forward.

🛡️

7 Validation Gates

Deterministic specd check — EARS syntax, design completeness, acyclic DAG, evidence, sync, traceability.

📉

DAG Wave Execution

Computes the concurrent runnable frontier so agents only touch tasks whose deps are resolved.

💾

Evidence-Gated Completion

specd verify runs the task's own command, records exit code + git HEAD. Complete only on a passing record.

🚦

Frontier Dispatch

specd dispatch emits ready-to-run packets for parallel subagents — role prompt + contract + verify.

🔌

Agent-Agnostic

Claude Code, Cursor, Aider — any command-running agent. Wired via a localized prompt pack + steering constitution.

2 · Why it exists — the problem it kills

Traditional AI coding

  • ❌ Free-form prompts, lost between sessions
  • ❌ Context window is the source of truth
  • ❌ "Trust me, it works" completion
  • ❌ Linear todo lists, no parallelism
  • ❌ Workflow re-invented per agent / per repo
  • ❌ Plan drift the moment context compacts

specd-driven coding

  • ✅ Structured spec artifacts on disk
  • ✅ Markdown files as the source of truth
  • ✅ Evidence-gated completion (exit code + HEAD)
  • ✅ DAG wave execution — parallel subagents
  • ✅ One agent-agnostic CLI contract
  • ✅ Steering constitution outlives chat sessions
The core insight: LLMs are excellent at reasoning and terrible at remembering process across a long, compacting context. So specd stops asking the model to police itself. The model proposes; a deterministic Go binary decides whether the gate opens.

4 · The command surface

Lifecyclespecd initScaffold .specd/ — steering, roles, skill pack, AGENTS.md
specd new <slug>Create a spec with 6 artifact stubs
specd approve <slug>Human gate — advance phase if gates pass
Executionspecd next <slug>Next runnable task (or --all for the frontier)

specd verify <slug> <id>Run task's verify command, record exit code + output tail + HEAD

Inspectionspecd status / waves / context / checkProgress board · wave graph + critical path · context briefing · run gates
specd report --format htmlSelf-contained HTML/MD snapshot — no LLM, deterministic

Programspecd program link/statusCross-spec DAG — which whole specs are runnable
Built for machines: SPECD_JSON=1 gives structured JSON on every command; specd help --json dumps the full command schema; exit codes are deterministic (0 ok · 1 gate/verify fail · 2 usage · 3 not found). All state mutations are atomic + versioned (advisory lock + compare-and-swap) — safe for concurrent agents.

5 · Real-world examples

Example A — Solo dev shipping a feature with Claude Code

$ specd init
$ specd new password-reset --title "Self-serve password reset"

# Agent authors requirements.md in EARS, then:
$ specd check password-reset
✓ EARS  ✓ design(skipped)  ...  ✗ traceability
$ specd approve password-reset        # requirements → design

# Agent fills design.md (7 headers), tasks.md (DAG), approves through.
$ specd next password-reset
T1 — Add reset-token table + migration   (role: builder)

# Agent implements T1, then specd — not the agent — proves it:
$ specd verify password-reset T1
running: go test ./...    exit=0  HEAD=a1b9f3c  (4.1s)
$ specd task password-reset T1 --status complete
✓ T1 complete (verify record exit 0)
Win: the agent cannot mark T1 done by claiming success. It completes only because a real test run exited 0 — recorded with the git HEAD it passed against.

Example B — Parallel subagent fan-out (delegate mode)

$ specd dispatch checkout-flow --json
# emits N packets, one per frontier task — each with rolePrompt + contract + files + verify
[
  { "id":"T3", "role":"builder",  "files":["cart.go"],     "verify":"go test ./cart/..." },
  { "id":"T4", "role":"builder",  "files":["pricing.go"],  "verify":"go test ./pricing/..." },
  { "id":"T5", "role":"investigator", "files":["*.go"],    "verify":"N/A" }
]
# Orchestrator spawns 3 subagents in parallel — no shared-file collisions,
# because the DAG guarantees these tasks have no inter-dependencies.
Win: true parallel execution. The frontier is the set of tasks whose deps are all resolved, so concurrent subagents never step on each other or on unfinished prerequisites.

Example C — Multi-spec program (a real release train)

$ specd program link api --on auth     # api waits for auth
$ specd program link web --on api      # web waits for api
$ specd program status
┌─────────┐   ┌─────────┐   ┌─────────┐
│  auth   │──►│   api   │──►│   web   │
│ Wave 1  │   │ Wave 2  │   │ Wave 3  │
└─────────┘   └─────────┘   └─────────┘
runnable now: auth
Win: the same DAG discipline that orders tasks inside a spec also orders whole specs across a program — an orchestrator knows exactly which spec is safe to start.

Example D — Mid-flight requirement change without losing the plot

$ specd midreq checkout-flow "Add Apple Pay" --impact high \
    --interpretation "new payment provider path" --changes "T4, +T7"
# Logged to mid-requirements.md, change-locked until human approves.
$ specd decision checkout-flow "Use provider-agnostic PaymentGateway iface" --supersedes ADR-2
Win: requirement churn is captured as an auditable record (impact + interpretation + ADR) instead of silently mutating the agent's memory.

7 · Strategies to work with specd

Strategy 1 — Tight inline loop (solo, default)

One agent, subagentMode: inline. Drive the ratchet by hand-feeding the agent the cycle: next → implement → verify → task complete. Best for small features and getting fluent with the gates.

Strategy 2 — Orchestrated fan-out (delegate)

An orchestrator agent runs dispatch --json, spawns one subagent per packet, monitors the frontier, dispatches the next wave when it clears. Best for large specs with wide, independent waves. Maximizes wall-clock throughput.

Strategy 3 — CI as the enforcer

Put specd check in CI as a required status. A PR that touches a spec must pass all 7 gates before merge — turning specd from a local discipline into an org-wide one. Exit code 1 = block merge.

Strategy 4 — Steering-first bootstrapping

Before the first spec, have the agent author the steering constitution (product.md, tech.md, structure.md) via the specd-steering skill, and set config.defaultVerify to the repo's real test command. Every later spec inherits this context for free.

Strategy 5 — Program-level release trains

Model a multi-service release as linked specs (program link). The orchestrator polls program status to know which spec is runnable next — DAG discipline scaled from tasks to deliverables.

Picking a mode: start at Strategy 1 to learn the gates, graduate to 2+3 once the team trusts the records, reach for 5 only when you genuinely have cross-spec dependencies — premature program modeling is overhead.

8 · Missing features & concepts — the roadmap to world-wide adoption

specd's core thesis (deterministic harness around a non-deterministic agent) is strong and under-served. To become the default harness for CLI agents, the gaps below matter most. Each idea is tagged by impact (adoption upside), effort (build cost), and moat (how hard to copy / how sticky).

Theme A · Frictionless adoption — the first 5 minutes
A1

Native MCP server (specd mcp)

impact: very higheffort: medmoat: high

Expose every command as Model Context Protocol tools so Claude Code / Cursor / any MCP host can call specd_next, specd_verify natively — no shelling out, no brittle prompt-pack parsing. This is the single biggest lever for "world-wide used with CLI agents."

Why: MCP is becoming the universal agent-tool interface. A first-class MCP surface turns specd from "a CLI an agent must be taught to drive" into "a tool the agent already knows how to use."

A2

One-shot scaffolding from a prompt (specd new --from "...")

impact: higheffort: lowmoat: low

Let the agent generate a draft requirements.md + design.md + tasks.md DAG from a natural-language feature description in one call, then iterate against the gates. Reduces time-to-first-green-check from ~20 min to ~2.

Why: The blank-stub cold start is the highest-friction moment. The gates already tell you what "valid" means — generate toward them.

A3

Editor/IDE extension + live HTML dashboard

impact: medeffort: medmoat: med

A VS Code panel rendering the live wave DAG, frontier, blockers, and gate status — auto-refreshing off state.json (the HTML report's autoRefreshSeconds hints this is intended). Humans see what the agent is doing without reading JSON.

Why: Adoption needs a human-facing surface. The deterministic report is already there; make it ambient.

Theme B · Closing the trust loop — better evidence
B1

Semantic acceptance gate (criteria ↔ test mapping)

impact: very higheffort: highmoat: very high

Today verify proves "tests passed." It does not prove the tests cover the acceptance criteria. Add a gate that maps each EARS criterion to ≥1 named test and fails if a criterion is untested. The gates.acceptance config key is a stub waiting for this.

Why: The deepest objection to AI code is "green tests, wrong behavior." Closing the criterion→test loop is specd's strongest possible moat — it makes "done" mean "the requirement is demonstrably satisfied," not "some test passed."

B2

Coverage & diff-scope evidence

impact: highmedmed

Have verify also capture coverage delta and assert the task's files: contract — flag when a builder touched files outside its declared scope. The record already holds HEAD; add changed-file set + coverage to it.

Why: Scope creep and untested branches are where agents quietly go wrong. The harness is the right place to catch both, deterministically.

B3

Verify sandboxing (container / namespace isolation)

impact: highhighhigh

Today verify runs sh -c as the user (env scrubbed, but real code exec on agent-authored input — SECURITY.md is explicit). Offer an opt-in sandboxed runner (container, bwrap, or seccomp) so untrusted tasks.md can be verified safely.

Why: "Only run verify on a tasks.md you trust" is a real adoption ceiling for autonomous fleets. Safe-by-default verify unlocks running specs you didn't author.

Theme C · Scale & orchestration — fleets of agents
C1

Long-running daemon + event stream (specd watch)

highmedmed

A daemon that watches the program DAG and emits frontier-change events (webhook / SSE / JSON-lines) so external orchestrators react instead of polling program status in a loop. Turns specd into the scheduler core of an agent fleet.

Why: Parallel orchestration today is poll-driven. An event stream is what production multi-agent systems need to scale past a handful of tasks.

C2

Distributed locking + remote state backend

medhighmed

The advisory file lock + CAS works locally. For agents on different machines sharing a spec, add a pluggable state backend (git-native, or Redis/Postgres) with the same CAS contract. Keeps the "atomic, versioned" promise across hosts.

Why: Cloud agent platforms run subagents on separate workers. Shared, consistent state is the precondition for distributed fan-out.

C3

Cost & telemetry ledger

medlowlow

Record per-task wall-clock, retries, verify duration, and (optionally) token/cost annotations into state. specd report then shows where time/money goes per wave — the missing analytics layer for agent work.

Why: Teams adopt what they can measure and justify to finance. specd already records the structure; add the meter.

Theme D · The discipline itself — smarter gates
D1

Automatic rollback on failed verify

highlowmed

Since verify records git HEAD, offer specd verify --revert-on-fail to auto-restore the working tree (or stash) when a task's verify fails — leaving the repo clean for the next attempt instead of half-applied.

Why: Failed agent attempts leave messy trees that poison the next task. The harness has the HEAD; it can guarantee a clean retry surface.

D2

Plugin / custom-gate API

highmedhigh

Let orgs register custom gates (license headers, ADR-required-for-arch-change, perf budgets) via a declared hook contract — same exit-code semantics. Ships the 7 core gates; lets the ecosystem add the 8th.

Why: Extensibility is how a tool becomes a platform. A gate API turns specd's discipline into a substrate others build standards on.

D3

Replay & spec diff / time-travel

medmedmed

specd replay <slug> reconstructs the full decision timeline (approvals, verifies, midreqs, ADRs) and specd diff shows how requirements/design evolved across phases. The records exist; surface them as a narrative.

Why: Post-mortems and onboarding need "how did we get here." specd already has the audit trail — make it readable.

Theme E · Ecosystem & standardization
E1

Spec-pack registry (shareable templates)

highmedhigh

A registry of reusable spec/steering/role packs (e.g. "Next.js feature", "Go microservice", "Rust CLI") installable via specd init --pack go-service. Network effects: the more packs, the more reason to start in specd.

Why: Templates are how frameworks win. A community of packs makes specd the obvious default instead of a blank-stub chore.

E2

The EARS/DAG spec format as an open standard

very highlowvery high

Publish the artifact schema (EARS requirements + 7-key task DAG + evidence record) as a documented, versioned open spec format with a JSON Schema. Let other tools read/write it. specd becomes the reference implementation of a standard, not just a tool.

Why: Tools get replaced; standards endure. If competitors interop with the specd format, specd has already won the category — this is the deepest moat on the page.

E3

GitHub-native integration (PR ↔ spec binding)

highmedmed

A GitHub App / Action that posts the wave DAG + gate status as a PR check and comment, links each commit to its task ID, and blocks merge on gate failure. Meets developers exactly where review already happens.

Why: Distribution beats features. Living inside the PR is how a workflow tool reaches millions without anyone running install.sh first.

9 · North star — the sequencing bet

If the goal is "the world-wide-used harness for CLI agents," the ideas above aren't equal. Recommended order:

  1. A1 — MCP server. Removes the single biggest adoption barrier: agents calling specd natively. Do this first.
  2. B1 — Semantic acceptance gate. The defensible moat. "Done means the requirement is demonstrably satisfied" is a claim no free-form agent can make.
  3. E2 — Open spec format. Standardize the artifact schema so the format outlives any one tool — turns specd into the reference implementation of a category.
  4. A2 + E1 — One-shot scaffolding + spec-pack registry. Kill the cold-start friction; seed network effects.
  5. C1 + E3 — Daemon/event stream + GitHub-native. Scale to fleets and meet developers in the PR where they already are.
The thesis in one line: specd already wins on process integrity. The road to ubiquity is frictionless entry (MCP), an undeniable trust claim (acceptance gate), and a format other tools must interop with (open standard). Build those three and the rest is distribution.
