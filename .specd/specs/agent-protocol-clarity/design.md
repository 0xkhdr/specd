# Design — agent-protocol-clarity

> Make what specd already guarantees legible, and stop shipping prose that
> contradicts the enforcement. Everything here is additive to existing output or
> corrective to existing text. No new verb, no new state machine.

- references: R1, R1.1, R1.2, R2, R2.1, R2.2, R3, R3.1, R3.2, R3.3, R4, R4.1, R4.2, R5, R5.1, R6, R6.1, R6.2
- disposition: accepted
- owner: maintainer

## Boundaries

- Owned: `internal/core/embed_templates/roles/`, a new role capability contract in
  `internal/core/roles.go`, the machine-output envelope shared by `status --guide`,
  `handshake`, `context --json`, and MCP results, plus the refusal shape they all return.
- Excluded: the gate registry's decisions, evidence and completion semantics, the DAG,
  orchestration leases, and anything requiring a host to enforce filesystem or shell scope.

## Interfaces

- `RoleCapability` — machine-readable per-role contract: effect set, allowed operations,
  completion authority, human authority, path scope, network policy, sandbox requirement.
  Role Markdown explains behavior; the contract alone defines authority (R6.1).
- `Assurance` — enumerated `advisory` | `gated` | `sandboxed`, computed from the active
  profile and the host's advertised sandbox support, carried on every machine response (R3.1).
- `Refusal` — stable code, blocker text, `authority_consumed`, `retry_safe`, `actor_required`,
  `recovery_command`; one shape for every refusal path (R4.1, R4.2).
- `Locator` — active spec, phase, status, revision, actor class, authority state, legal next
  operations, human-only boundary, authoritative flag; embedded in every machine response (R5.1).
- `specd request-decision <slug> <task> --reason --proposed-change --impact` — the agent-legal
  operation the craftsman role points at instead of the human-only `specd decision` (R1.1). It
  suspends the task and creates a human action; it records no approval.

## Invariants

- Determinism holds: assurance, refusal, and locator values are pure functions of on-disk
  `.specd/` state plus host-declared capabilities. No LLM in any of these paths.
- `request-decision` creates a pending human action and never advances a phase, completes a
  task, or writes evidence.
- Additive-only machine fields; no existing field changes meaning or type.
- Authority derives solely from capability contracts. Markdown that disagrees is a test
  failure, never a runtime resolution (R6.2).
- Assurance never upgrades implicitly; an unrecognized stored level degrades to advisory.

## Failure

- Host misreports sandbox support: assurance is a claim about the host, so `sandboxed` is
  recorded together with the host identity that asserted it and the session stays labeled.
- A refusal path missed during the migration returns an untyped error: contained by a test
  that enumerates refusal sites and asserts every one returns the structured shape.
- `request-decision` used to stall work indefinitely: the suspended task remains visible as a
  human-owned blocker in `status --guide`, so the stall is reported rather than silent.

## Integration

- MCP and CLI project the same palette, so the envelope is added once at the machine-response
  boundary and both surfaces inherit it.
- `docs/command-reference.md` is generated by `tools/gendocs`; adding `request-decision` to
  `internal/core/commands.go` requires regenerating it or `scripts/docs-lint.sh` fails CI.
- Existing adapters continue to parse older fields unchanged; new fields are ignorable.

## Alternatives

- Let the craftsman role keep pointing at `specd decision` and rely on the agent to escalate to
  a human out of band — rejected: it makes a denied command the documented happy path.
- Express assurance as a boolean `governed` flag — rejected: it collapses the advisory/gated
  distinction that operators most need, since gated enforcement is real but not host-enforced.
- Encode role authority in the role Markdown frontmatter — rejected: prose and authority must
  not share a source, or editing documentation silently edits permissions.

## Verification

- R1.1/R1.2: test that parses every shipped role and steering file for referenced commands and
  asserts each is permitted by that role's capability contract.
- R2.1/R2.2: test asserting each role's contract declares an explicit effect set and that the
  validator declares `workspace-read` and `harness-evidence-write`.
- R3.1/R3.2: test asserting every machine response carries an assurance level, and that a host
  without advertised sandbox support yields `advisory`.
- R3.3: `docs/agent-integration.md` states the default/production enforcement boundary.
- R4.1/R4.2: table test over every refusal site asserting the structured shape and populated fields.
- R5.1: test asserting the locator block is present and complete on each machine surface.
- R6.1/R6.2: conformance test asserting role Markdown and capability contract agree.

## Deployment

- Ships in one binary release; no migration of `.specd/` state is required because every change
  is additive to output or corrective to embedded templates.
- Managed role and steering regions update through `specd init --refresh`, previewed with `--dry-run`.
- Observation: `specd status <slug> --guide --json` shows assurance and locator fields.
- Ownership: maintainer.

## Rollback

- Trigger: an adapter breaks on the new envelope, or `request-decision` is found to create a
  state a human cannot clear.
- Path: revert the release; additive fields disappear and older parses are unaffected. Any
  pending `request-decision` records are inert data under `.specd/` and block nothing once the
  verb is absent.
